## 1. Purpose

`my-mastra-app/src/mastra/agents/concierge.ts` defines `lastIntent` as an enum of `['rental_search', 'event_discovery', 'chitchat', 'unknown']`. The concierge already wires `searchRestaurantsTool` and `searchAttractionsTool` (lines 5–6), but those intents have nowhere to be persisted in working memory. Result: a follow-up like "any cheaper restaurants?" can't reuse the prior restaurant query and the agent re-clarifies, hurting multi-turn quality.

## 2. Goals

- Working memory enum carries every intent the concierge can act on.
- Add `lastRestaurantQuery` and `lastAttractionQuery` shapes alongside the existing `lastRentalQuery` / `lastEventQuery`.
- Studio working-memory tab shows the new fields after a restaurant search turn.

## 3. Features

- Restaurant + attraction follow-ups (`"any cheaper"`, `"closer to Laureles"`, `"free options"`) reuse prior filters instead of re-asking.

## 4. Workflow

1. Update enum to `['rental_search', 'event_discovery', 'restaurant_search', 'attraction_search', 'chitchat', 'unknown']`.
2. Add `lastRestaurantQuery: { neighborhood, cuisine, maxPricePerPerson }` and `lastAttractionQuery: { neighborhood, freeOnly, category }` to the schema (mirror `lastRentalQuery` shape).
3. Update concierge instructions to set these fields after a restaurant or attraction tool call.
4. Verify in Studio: run "Italian restaurants in Laureles under 30", then "any cheaper" — second turn should reuse `cuisine='italian'` + `neighborhood='Laureles'`.

## 5. User journeys

- **Camila** asks for restaurants, then narrows by price — concierge keeps the cuisine.
- **Andrés** asks for free attractions in Comuna 13, then "anything for kids" — concierge keeps `freeOnly=true`.
