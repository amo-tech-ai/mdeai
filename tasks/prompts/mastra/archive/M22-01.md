# M22-01 — Smoke script must build temp env file when secrets come from process env

> **Summary:** The smoke script falls back to `.env` when Postgres is unreachable, but secrets injected by Infisical live in process env — not in `.env`. Fix: always write a `mktemp` temp file from process env, never point `--env` at `.env`.
>
> **Verification:** Bug confirmed on `origin/main` at `mastra-smoke.sh:121-126` via `git show`. Fix already applied in PR #23 (`fix/m22-01-smoke-env`, open).
>
> **Status:** 🟡 In Progress · PR #23 open · Ready to review and merge
>
> **Key docs:** [Mastra Studio](https://mastra.ai/docs/studio/overview) · [Mastra Server](https://mastra.ai/docs/server/mastra-server) · [mastra-smoke-test skill](../../../../.claude/skills/mastra-smoke-test/SKILL.md) · [mde-infisical skill](../../../../.claude/skills/mde-infisical/SKILL.md)
>
> **Gate commands:**
> ```bash
> infisical run -- npm --prefix my-mastra-app run smoke:runtime   # must exit 0
> ls /tmp/mdeai-mastra-smoke-env.*  # must return nothing (cleanup verified)
> ```

## 1. Purpose

`my-mastra-app/scripts/mastra-smoke.sh` forces `SMOKE_ENV_FILE=".env"` when the local Postgres fallback is unreachable. If secrets come from process env (e.g. `infisical run -- npm run smoke:runtime`), pointing `mastra dev --env .env` at a thin or stale `.env` file means the dev server boots without OpenAI / Gemini / Infisical-injected keys, and the runtime smoke step fails with `OPENAI_API_KEY not found`. This blocks CI and any clean local repro.

## 2. Goals

- Smoke script never points `--env` at `.env` when secrets are present in the parent process env.
- `infisical run -- npm run smoke:runtime` succeeds end-to-end on a fresh worktree with no `.env` file present.
- Cleanup deletes the temp env file on `EXIT`.

## 3. Features (what the user gets)

- One command (`infisical run -- npm run smoke:runtime`) brings up Mastra Studio with the right secrets, exercises the runtime path, and tears down cleanly.
- Local devs without Infisical can still use `.env` exactly as today.

## 4. Workflow

1. In the script's `else` branch (around `mastra-smoke.sh:121-126`), always create `mktemp` instead of falling back to `.env`.
2. Write `OPENAI_API_KEY`, `GOOGLE_GENERATIVE_AI_API_KEY`, `MODEL`, `DATABASE_URL` (plus any other vars `env_value` finds) into the temp file.
3. Keep the existing `chmod 600` and `EXIT` cleanup.
4. Add a log line that names the source of each var (`process-env` vs `dotenv`).

## 5. User journeys

- **Engineer running CI smoke** — runs `infisical run -- npm run smoke:runtime`, sees green output, no leftover env file in `/tmp` after exit.
- **Engineer with only `.env`** — same command works without Infisical because `env_value()` reads `.env` for each key.

## 6. Agents

None — this is a shell-script and CI plumbing change.

## 7. Integrations

| Integration | Purpose | Auth source |
|---|---|---|
| Infisical | Inject `OPENAI_API_KEY`, `GOOGLE_GENERATIVE_AI_API_KEY` | parent process env |
| Mastra dev server | Read secrets via `--env <file>` | tmp file written by script |

## 8. Summary

Stop forcing `.env` for the dev server when secrets live in process env. Build a per-run temp file instead. Verified by `infisical run -- npm run smoke:runtime` going green on a fresh checkout.
