# M22-05 — Constrain agent working-memory price fields to positive numbers

> **Summary:** Tool input schemas already use `z.number().positive()` at the boundary, but agent working-memory schemas use plain `z.number().optional()` for price fields. A model-induced negative value (`"set budget to -50"`) can be persisted and silently surfaces as an absurd cap in later turns. Fix: add `.positive()` to every price field and `.int().nonnegative()` to bedroom counts in working-memory schemas.
>
> **Verification:** Bug confirmed on `origin/main` — `grep -n "maxPricePerNight\|maxPricePerTicket" my-mastra-app/src/mastra/agents/concierge.ts` shows `z.number().optional()` with no `.positive()` guard.
>

## 1. Purpose

Tool input schemas already use `z.number().positive()` (e.g. `search-events.ts:108`), so the boundary is safe. But the *agent working-memory* schemas use plain `z.number().optional()` for `maxPricePerTicket` (event-agent), `maxPricePerNight` (concierge), `maxPricePerPerson` (restaurant). A model-induced negative value can be persisted into working memory and surface in later turns as an absurd cap.

## 2. Goals

- All price fields in agent working-memory schemas use `z.number().positive().optional()`.
- Zod parse failure is logged, not crashed.

## 3. Features

- Working-memory inspector in Studio never shows a negative price.

## 4. Workflow

1. Audit every `z.number()` in `my-mastra-app/src/mastra/agents/*.ts` working-memory schemas.
2. Constrain prices with `.positive()`. Constrain `minBedrooms` with `.int().nonnegative()`.
3. Run `mastra dev`, exercise concierge with a pathological prompt ("set my budget to -50"), verify working memory rejects gracefully.

## 5. User journeys

- **QA** — works through edge prompts; agent never persists junk values.
