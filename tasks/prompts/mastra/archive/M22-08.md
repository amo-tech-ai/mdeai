# M22-08 — Align rental card field names: `sourceUrl` vs `source_url` mismatch

> **Summary:** The concierge prompt template uses `<source_url>` (snake_case) but the `cardSchema` emits `sourceUrl` (camelCase). The model renders the placeholder literally — Camila sees "View listing → `<source_url>`" instead of a real URL. Fix: update the prompt to use `<sourceUrl>` matching the schema.
>
> **Verification:** Bug confirmed on `origin/main` — `concierge.ts:102` has `<source_url>`; `rental-search-workflow.ts:39` cardSchema has `sourceUrl: z.string()`. These do not match.

## 1. Purpose

`my-mastra-app/src/mastra/agents/concierge.ts` lines 102–103 instructs the agent to render
```
View listing → <source_url>
Schedule viewing → <schedule_viewing_url>
```
But `my-mastra-app/src/mastra/workflows/rental-search-workflow.ts` line 90–91 emits cards with `sourceUrl` and `scheduleViewingUrl` (camelCase). The model dutifully renders `<source_url>` literally because the field doesn't exist on the card. Camila clicks a broken placeholder.

## 2. Goals

- Concierge prompt + cardSchema + workflow emit use **one** name. Recommendation: keep camelCase (`sourceUrl`, `scheduleViewingUrl`) — TypeScript convention.
- Studio test: rental-search returns cards whose URLs render correctly in the chat reply.

## 3. Features

- Working "View listing" / "Schedule viewing" links in concierge replies.

## 4. Workflow

1. Pick a single naming: keep `sourceUrl` / `scheduleViewingUrl` (matches workflow + TS convention).
2. Update `concierge.ts` prompt (around line 102–103) to use `<sourceUrl>` and `<scheduleViewingUrl>`.
3. Grep for any other reference to `source_url` / `schedule_viewing_url` in `my-mastra-app/src/`.
4. Studio test: ask "1BR Laureles under $80", check the reply contains real URLs not the placeholder text.

## 5. User journeys

- **Camila** — clicks `View listing` and lands on the host page, not at `<source_url>`.
